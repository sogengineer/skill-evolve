# Coach Meta Prompt(汎用雛形)

Step -1 pilot で確定させる `coach_meta_prompt.md` の汎用雛形。
**コーチ subagent は毎イテレーション、このメタプロンプトを最優先で読む。**

> 共通の質的判断シグナル基準(数値付きヒューリスティック、強制 4 項目、自己診断チェック)は
> SKILL.md 本体の「コーチ subagent 起動契約」節に集約されている。
> 本ファイルは **ドメイン依存の前提とプロジェクト固有の禁止事項** だけを書く。

---

## 1. ドメイン前提(pilot で埋める)

- **対象スキルの役割**: <例: コードレビュー / 要約 / 検索 / 文書生成 etc.>
- **期待される出力スタイル**: <例: テーブル形式の指摘 / Markdown / JSON etc.>
- **読み手・実行者**: <例: シニア開発者 / 新人エンジニア / 別 LLM agent>
- **「正解」の典型像**: <例: 偽陽性ゼロ + 該当箇所が file:line で示される etc.>
- **避けたいスタイル**: <例: 抽象的なベストプラクティス羅列 / 過剰な保険的注釈 etc.>

---

## 2. プロジェクト固有の禁止事項

SKILL.md 本体の共通禁止事項(frontmatter 改変、rubric 自体への言及、全書き換え 等)に加えて、
本対象スキルでは次を恒久的に禁止する:

- <例: 「念のため X を確認すること」系の保険的追加は禁止(対象スキルは既に簡潔さを優先する設計)>
- <例: バージョン番号や日付の固定値を埋め込む編集は禁止(自動的に陳腐化する)>
- <例: コマンド例を追加する場合は必ず期待出力もセットで(コマンド単独追加は禁止)>

(初回は空でよい。permanent buffer 昇格や Phase D レビューでパターンが見えたら追記する)

## 4. ドメイン固有の rubric 解釈ヒント(任意)

feedback_spec.md に書かれた rubric の判定文言について、コーチが誤読しやすいニュアンスがあれば
ここに 1〜3 行で書く。

例:
- 「actionable」とは「ファイルパス + 行番号 + 該当コード断片」の 3 点セットが揃っていること
- 「偽陽性ゼロ」は規約からの逸脱のみを FAIL に上げることを指す。技術的好みでの FAIL は対象外

(これがあると、コーチが軸名から推測した修正を出しにくくなる)

---

## 5. 自己診断補足(任意)

SKILL.md 本体の自己診断チェックに加えて、本対象スキル固有のチェック項目があれば追記する。

例:
- [ ] 編集案が <ドメイン固有の慣習> に違反していないか
- [ ] <対象スキル名> の他セクションとの整合性が崩れていないか

(初回は空でよい)
